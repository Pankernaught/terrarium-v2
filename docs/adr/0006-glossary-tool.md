# Technical-term glossary is one id-keyed data file with author-controlled prose links

The app forces a wall of horticultural jargon in front of the user — controlled-vocab chips (`bright-indirect`, `montane`, `rosette`, `leca`), structural concepts (epiphyte, false bottom, paludarium), and anatomy terms in care prose (`rhizome`, `frond`) — with no explanation. We are adding a glossary that is both a browsable reference and a source of inline tap-to-define links.

**The data model: one `src/data/glossary.json`, keyed by canonical id** — vocab terms reuse the *exact* ids that already exist in `src/types/plant.ts` (`LIGHT_LEVELS`, `PLANT_TYPES`, `NATIVE_BIOMES`, …) and `src/data/substrate-components.ts`. Entries are grouped, and groups are of two kinds:

- **Enum-backed groups** (light, moisture, ph, growthRate, growthHabit, plantType, biome, substrate) are **coverage-checked**: a test loops each controlled vocabulary and asserts every value has an entry, failing CI on a miss — the same "unknown tag fails CI rather than silently shipping" idiom the substrate loader already uses. This is what keeps the glossary from drifting out of sync with the chips it explains.
- **Free-form groups** (`concept`, `anatomy`) have no enum and no coverage check — the curator authors slugs and display terms freely. This is the open-ended dictionary for tiers 2–4.

Keying by id (rather than a flat standalone file with its own keys) is what makes the **inline-linking half nearly free**: a rendered chip already holds its id (`plant.light.primary` *is* `'bright-indirect'`), so linking is a direct `lookupTerm(id)` with no string-matching, normalization, or parallel key table. Co-locating definitions inside the vocab files (a `description` on each enum value) was rejected: it would force the flat `as const` enums into object arrays — an invasive refactor of the type/engine layer for a presentation feature — and would scatter the corpus across 6+ files, making the single browsable screen harder to assemble.

**Prose links use author-controlled `[[slug|display]]` markup, not auto-scanning.** Seed `notes`/`nativeContext` mark exactly which mentions link (`keep out of [[direct]] light`, `spreads by [[rhizome|rhizomes]]`). A ~15-line pure parser splits on `[[…]]`; the optional `|display` form handles grammar (plurals/tense) without changing the slug. Auto-scanning the prose against the full glossary was rejected: it is genuinely ambiguous (everyday "low **light**", substring hits like "**moss**y"), needs a stop-list + plural rules + a test corpus, and runs on a hot render path — large effort and real false-positive risk for the *narrowest* gain (only anatomy terms that appear solely in prose, all of which remain reachable via Terms-mode search anyway). The `[[ ]]` convention deliberately mirrors the memory system's link syntax. A seed-time + test integrity check asserts every bracketed slug resolves, so a typo'd link fails CI instead of dead-linking at runtime.

**Surfaces:** the glossary lives in the **Browse** screen behind a "Plants | Terms" mode toggle (no new tab; a glossary does not earn permanent nav real estate), with category-filter chips reusing the existing `FacetGroup`/`Chip` components. A single `TermSheet` built on `BottomSheet` (sibling of `PlantSheet`) serves both list-taps and inline links, with `seeAlso` cross-links swapping content via the same `lastSlug`-reset pattern as `plant-sheet.tsx`.

**Layering** respects the CI-enforced import-purity rule: `filterGlossary` and `parseGlossaryMarkup` are pure modules in `src/logic/` (data passed as args, unit-tested); `loadGlossary`/`lookupTerm` + integrity checks live in `src/data/`; `TermSheet` in `src/components/`. Definitions are 2–4 sentences (definition + a terrarium-specific "why it matters"), LLM-drafted and curator-reviewed, no source citations in v1.

**Authoring** happens in the existing `tools/plant-admin/` tool (ADR 0005), extended with a glossary editor: a "Terms" mode + `/api/glossary` CRUD routes writing `glossary.json`, a form from a `GLOSSARY_FIELD_SPEC` in `field-spec.js`, a missing-coverage worklist (vocab values lacking an entry), and `[[slug]]` validation on save so bad prose links flag in-tool rather than only in CI.

Corpus boundary for v1 ≈ 70–80 entries: ~52 enum-backed (the coverage test is the backlog) + ~12 hand-listed concepts + a grep-derived set of anatomy terms that actually appear in the plant prose (`src/data/plants.json`, ~243 entries under `.plants`).
